# 海外 AI 陪伴产品研究工作台

本目录用于沉淀可复核的竞品研究、用户反馈、截图证据与产品决策。研究采用“飞书协作 + Git 快照”模式：

- **飞书 Docx**：多人共同编辑的叙事报告。
- **飞书 Base**：产品、证据、评论与任务的结构化主数据。
- **飞书 Markdown**：与 Git 对齐的阶段文档镜像，供团队直接查看和评论。
- **Git**：调研快照、采集脚本、方法模板和允许入库的证据文件。

截至 2026-07-17，飞书工作台文件夹、协作版 Docx 和研究 Base 已创建并完成字段核验。规模扫描 B001–B004 已沉淀 101 个产品；B005 已筛出 26 个重点产品和 8 个标杆，并推进至 Wave 3。当前本地快照共有 205 条证据、75 条评论样本、21 个候选种子和 17 项研究任务；18 个非标杆重点产品的轻量标准卡已完成 18/18，并完成首轮横向复核。Replika/Kindroid 已完成首日与短期回归，Character.AI/Wysa 已完成首批评论样本，ElliQ/Moflin 已完成官方证据与独立实测首轮。远端链接与 token 仅保存在被 Git 忽略的 `.lark-research-workspace.json`。

## 飞书文档布局

飞书研究工作台当前提供 32 份原生 Markdown 镜像：

- 根目录：主报告、研究工作台说明、D1–D4 产品方向团队评审单页、18 产品横向复核、轻量标准卡索引、关系生命周期深拆，共 6 份；

## 规模扫描漏斗

本轮不把“找到 100 个名称”当成交付，而采用可逐层复核的漏斗：

| 层级 | 目标数量 | 最低交付 |
| --- | ---: | --- |
| 候选池 | 300–500 | 产品名、平台标识、发现来源、初步活跃信号 |
| 标准卡 | 100–150+ | 开发者、平台 ID、规模口径、来源、核验日期、数据缺口 |
| 重点产品 | 20–30 | 核心机制、定价、关键流程截图、评论主题 |
| 标杆深拆 | 5–10 | 首日/首周、关系生命周期、记忆、主动触达、订阅墙与安全边界 |

规模入选优先使用 Android 公开安装区间；iOS 不公开下载量，评分数只能用于发现和排序，不能冒充下载量。Web-only 与硬件分别使用月访问量、公开用户数或出货量等独立口径。

本地快照校验命令：

```bash
python3 scripts/research/validate_catalog.py
```

飞书研究操作统一通过受限代理执行，以复用一次性系统授权，并避免 token 出现在动态 shell 命令中：

```bash
python3 scripts/research/lark_research.py --check
python3 scripts/research/lark_research.py base +table-list --offset 0 --limit 10
python3 scripts/research/lark_research.py docs +fetch --scope outline --max-depth 2
python3 scripts/research/lark_research.py markdown +diff --file docs/research/overseas_ai_companion_landscape.md
```

该代理只开放当前研究需要的 Base、Docx、Markdown 与 Drive 非破坏性操作；删除资源、修改共享权限/所有者和 Docx 全文覆盖默认禁止。

## 协作流程

1. 在 Base 的“研究任务”表认领任务，明确产品、问题和交付物。
2. 先把来源写入“证据库”，至少包含 URL、采集日期、证据类型和核验状态。
3. 截图写明页面/步骤、地区、版本和时间；动态页面同时保存 URL 与观察说明。
4. 评论先进入“用户评论”，按统一主题编码；不要直接从少量评论推导总体比例。
5. 结论进入 Docx 报告，并关联可追溯的证据或 Base 记录。
6. 关键结论由另一位参与者复核后，再标记为“已核验”。
7. 形成路线选择时更新“决策日志”，记录依据、负责人和复审条件。
8. 进入 Git commit 前检查飞书远端变更，合并后再刷新本地快照和 Markdown 镜像。

## 证据等级

| 等级 | 定义 | 典型来源 | 可以支撑什么 |
| --- | --- | --- | --- |
| A | 第一方、可直接核验 | 官方产品页、帮助中心、条款、商店页 | 功能、价格、产品定位等事实 |
| B | 独立且有直接体验或方法说明 | 权威媒体实测、研究论文、公开数据集 | 独立体验、效果或行业判断 |
| C | 可追溯的用户声音 | App Store、Google Play、公开社区评论 | 问题发现、主题假设，不代表总体比例 |
| D | 二手线索或厂商效果主张 | 聚合文章、无方法的市场数字、厂商案例 | 只能作为继续核验的线索 |

所有带数字的结论必须记录口径、地区、时间、样本量和来源。厂商自述、媒体体验和用户评论不能混为同一证据层级。

## 状态定义

- `待采集`：只有研究问题，尚无证据。
- `采集中`：已有来源或截图，但未完成交叉核验。
- `待复核`：分析完成，等待另一位参与者检查。
- `已核验`：来源、口径和结论均完成复核。
- `有争议`：证据冲突或结论尚未收敛。
- `已归档`：不再影响当前决策，但保留历史。

## 数据与隐私

- 默认不保存无关用户名、头像、联系方式等个人信息。
- 用户评论只保留研究所需内容、来源链接和上下文；对外引用避免大段复制。
- 遵守平台条款、robots 规则、访问频率和版权限制。
- 含账号、密钥、飞书 token 或未公开个人数据的文件不得提交 Git。
- 大体积视频、音频和批量截图优先存入 Base 附件或受控云空间，Git 仅保留索引与必要的小型证据。
