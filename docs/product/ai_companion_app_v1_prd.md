## 0. 结论摘要

V1 的目标不是把微信页面搬进 App，而是建立一条独立、可持续演进的 App 渠道：用户仅用手机号即可注册或登录，无需微信扫码，随后可进行文字聊天，也可以录音转成可编辑文字后发送；AI 在 V1 始终回复文字。

V1 的产品和技术边界已经冻结，并已完成首个可构建版本。当前状态如下：

| 能力 | 当前状态 | V1 结论 |
| --- | --- | --- |
| RN 客户端 | 已完成 Expo/React Native 双端工程、登录、聊天、录音转写和设置页 | iOS Simulator 已启动验证；Android 实体机已完成安装和手机号登录 |
| 手机号认证 | 已完成 App-first 注册/登录、阿里云验证码 WebView、OTP、30 天 Session 和退出撤销 | 待真实短信/验证码生产配置和双端真机验证 |
| 对话核心 | 已增加独立 `app` channel 和 `__app_active__` scope | App/微信短期会话隔离，账号与长期记忆复用 |
| 文字聊天 API | `/api/v1` 认证、历史、turn、幂等接口已完成 | 已通过聚焦回归和真实 HTTP 冒烟，待 staging/生产路由 |
| 语音输入 | 客户端录音和 provider-neutral 批量 ASR 已完成 | 本地 mock 已打通；真实 provider、质量基线和真机矩阵待定 |
| HTTPS 路由 | 本地接口闭环及仓库 Nginx `/api/v1/*` 配置已完成 | 配置尚未部署；需提供 staging HTTPS 并执行发布前预检 |
| 合规与上架 | 已有协议勾选、麦克风用途文案和安全存储 | 专用协议 URL、账号注销、正式签名和商店合规仍是公开发布门禁 |

因此，V1 仍由“RN 客户端 + App 渠道后端适配 + 批量 ASR + 发布基础”四部分组成。前三部分已具备内测代码闭环，发布基础需要在真实服务和真机环境完成。

- iOS：Release Simulator 构建成功，并已在 iPhone 15 Simulator 安装启动，无红屏或白屏。
- Android：Release APK 构建成功，包名 `com.ai4all.companion`、版本 `0.1.1`、versionCode 4，包含 ARM/x86 四种架构并通过 APK v2 签名校验；已在 Android 实体机完成安装和手机号登录。

## 1. 背景与目标

### 1.1 背景

现有 AI 陪伴服务主要通过微信扫码接入，在微信私聊中完成陪伴式聊天。App 的价值是建立不依赖微信的自有入口，为更稳定的身份体系、可控的聊天体验，以及 V2 实时语音通话打基础。

### 1.2 V1 产品目标

V1 必须完成以下最小闭环：

1. 用户使用中国大陆手机号完成注册或登录，不需要绑定微信。
2. 用户注册后可直接进入一个单 Agent、单对话流的聊天界面。
3. 用户可发送文字，可靠获得 AI 的文字回复。
4. 用户可录制语音，将转写结果放入输入框，编辑确认后按普通文字发送。
5. 已有微信用户使用同一手机号登录时，复用原有 AI4ALL Account 和长期记忆，不创建重复账号。
6. 为 V2 实时语音保留独立 App 渠道、鉴权、音频权限、服务端凭证签发和实时会话边界。

### 1.3 成功定义

V1 上线后，一个从未使用过产品的新用户可以独立完成“手机号验证 → 进入聊天 → 发出首条文字或语音转写消息 → 收到文字回复”，全程不依赖微信。

## 2. 用户与使用场景

### 2.1 目标用户

- 希望随时倾诉、获得温和回应或轻量陪伴的普通用户。
- 已在微信使用 AI4ALL，希望改用独立 App 的存量用户。
- 首发地区默认中国大陆，首发认证仅支持 `+86` 手机号。

### 2.2 高频场景

- 碎片时间发一段文字，继续之前的话题。
- 不方便打字时说一段话，确认转写后发送。
- 更换设备后通过手机号重新登录并恢复 App 对话历史。
- 微信与 App 交替使用，同一个 AI 保留长期偏好，但两端近期消息流互不混杂。

## 3. 产品原则与关键决策

### 3.1 已按本 PRD 采用的决策

| 编号 | 决策 | V1 取值 |
| --- | --- | --- |
| D1 | 注册模式 | App-first；手机号注册后可直接聊天，不要求微信扫码 |
| D2 | 新老用户识别 | 注册与登录合一；同一手机号命中已有 `platform_user` 时直接登录 |
| D3 | 账号复用 | 同一用户复用最早创建且有效的主 `aid_` 账号；客户端不传 `account_id` |
| D4 | 渠道模型 | 新增独立 `app` 渠道，不借用 `web` 或 `openclaw-weixin` |
| D5 | 会话与记忆 | App/微信短期会话隔离；Soul、用户偏好和长期记忆按账号共享 |
| D6 | 首次体验 | 使用默认 AI 设定直接可聊；轻量认识流程不得阻断用户真实问题 |
| D7 | 语音形态 | V1 是“语音转文字输入”，不是发送语音消息，也不是实时通话 |
| D8 | ASR 形态 | 停止录音后上传到后端做批量 ASR；转写进入输入框，用户可编辑后发送 |
| D9 | AI 输出 | V1 仅文字回复，不做 TTS、语音消息或实时音频下行 |
| D10 | RN 工作流 | 优先采用支持原生模块的 TypeScript Bare/Prebuild 工作流，避免 V2 再迁移 |
| D11 | 移动端会话 | V1 使用服务端可撤销的 30 天 opaque token；暂不增加 refresh endpoint |
| D12 | 发布方式 | 首包用于受控内测；公开上架前必须完成正式签名、注销与合规门禁 |
| D13 | 产品标识 | 产品名“朝夕相伴”，bundle id/package 为 `com.ai4all.companion` |

### 3.2 体验原则

- 陪伴优先：界面像私人对话，不像客服工单或模型调试工具。
- 首次即可用：不通过多页资料表阻塞首聊。
- 用户可控：ASR 结果先可见、可改，再发送；不在识别错误时替用户发言。
- 不丢不重：网络超时后重试同一条消息，不重复消耗模型、不出现双回复。
- 隐私克制：服务端不长期保存原始录音，埋点不记录聊天正文和手机号明文。

## 5. 核心用户流程

### 5.1 新用户注册并首聊

1. 用户打开 App，阅读并勾选用户协议和隐私政策。
2. 输入 `+86` 手机号，完成人机验证。
3. 获取并输入 6 位短信验证码。
4. 服务端验证通过，创建 `platform_user`、默认 `aid_` 账号和 App 会话凭证。
5. App 安全保存凭证，进入聊天页。
6. 聊天页展示服务端下发的默认 AI 名称、头像和欢迎语。
7. 用户发送文字或先用语音生成文字。
8. AI 必须先回应用户的真实内容；可在合适时机轻量询问称呼或陪伴偏好。

### 5.2 老用户登录

1. 用户完成相同的手机号 OTP 流程。
2. 服务端识别已有 `platform_user`，选择其有效主账号。
3. 若该账号此前来自微信，保留已有 Soul、偏好和长期记忆。
4. App 只展示 App 当前会话历史；首次进入 App 时历史可为空。
5. 用户可直接继续聊天，不要求重新扫码或重复配置 AI。

### 5.3 文字聊天

1. 用户输入非空文本并点击发送。
2. 客户端生成稳定的 `client_message_id`，立即显示待发送气泡。
3. 服务端在 App conversation scope 内执行同一套陪伴对话核心。
4. 等待期间显示“正在回复”，本轮禁止重复提交。
5. 成功后展示 AI 文字气泡；失败时原气泡保留并提供“重试”。
6. 超时重试必须复用原 `client_message_id`；服务端返回已完成结果，不重复执行。

### 5.4 语音转文字输入

1. 用户点击麦克风，首次使用时触发系统麦克风授权。
2. 授权后开始录音，界面显示时长、取消和停止操作。
3. 用户停止录音后，客户端上传音频并进入“正在识别”。
4. ASR 成功后把文字放入输入框，不自动发送。
5. 用户可修改、清空或点击发送；发送后完全走文字聊天链路。
6. ASR 失败、无有效语音或超时时，保留输入框原内容并给出重试/改用键盘提示。

## 7. 页面与状态

### 7.1 页面清单

1. 启动/登录态恢复页。
2. 手机号登录页。
3. 人机验证承载层（原生 SDK 或受控 WebView）。
4. 短信验证码页。
5. 聊天主页。
6. 最小设置页。
7. 账号注销确认页（公开版）。

### 7.2 聊天页布局

```text
┌──────────────────────────────┐
│ [头像] AI 名称          设置 │
│        一句轻量人设说明       │
├──────────────────────────────┤
│                              │
│ [AI] 今天过得怎么样？         │
│                              │
│          有点累，刚下班 [我]  │
│                              │
│ [AI] 正在回复…                │
├──────────────────────────────┤
│ [麦克风] [输入消息…]   [发送] │
└──────────────────────────────┘
```

键盘弹起时输入区必须保持可见；消息列表保持当前位置，首次进入和新消息到达时按规则滚动到底部。

### 7.3 关键异常状态

| 状态 | 用户提示与动作 |
| --- | --- |
| 无网络 | 显示离线状态；保留草稿；允许用户手动重试 |
| 登录过期 | 清除失效凭证并回到手机号验证；不得继续展示新拉取的隐私数据 |
| OTP 过频 | 展示服务端可重试时间，不持续触发人机验证 |
| turn 进行中 | 保持等待或短暂退避，用同一 `client_message_id` 查询/重试 |
| turn 超时 | 显示“回复仍可能在生成”，用同一 id 重试并回拉历史 |
| 达到限流 | 展示可理解文案，不自动高频重试 |
| 麦克风拒绝 | 解释用途，可跳转系统设置，也可继续键盘输入 |
| ASR 失败 | 不发送空文本；允许重新录音或键盘输入 |
| 服务维护 | 展示统一维护态和稍后重试入口 |

## 8. 后端接口契约建议

以下为 V1 已实现并冻结的客户端契约；RN 不应直接依赖 `/web/*`、`/openclaw/*` 或内部 DB 结构。V1 Session 固定为 30 天 opaque token，因此不提供 refresh endpoint。

### 8.1 通用约定

- 公网前缀：`/api/v1`。
- 业务响应携带 `request_id`，错误使用稳定 `code`，展示文案可由客户端本地化。
- 登录后使用 `Authorization: Bearer <access_token>`。
- 所有用户数据响应设置 `Cache-Control: no-store`。
- 客户端不提交 `account_id`；服务端只根据登录用户解析其主账号。
- 手机号、token、聊天正文和音频不得出现在普通访问日志。

### 8.2 接口清单

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| GET | `/api/v1/app/config` | 公共配置、验证码参数、功能开关、最低版本 |
| POST | `/api/v1/auth/otp/send` | 人机验证后发送 OTP |
| POST | `/api/v1/auth/otp/verify` | 校验 OTP，返回一次性 verified token |
| POST | `/api/v1/auth/session` | 注册/登录合一，返回 App 会话与账号摘要；不得启动微信扫码 |
| DELETE | `/api/v1/auth/session/current` | 退出并撤销当前设备会话 |
| GET | `/api/v1/me` | 当前用户、主账号、AI 展示信息和 onboarding 状态 |
| DELETE | `/api/v1/me` | 发起账号注销（公开版待实现，不属于当前受控内测包） |
| GET | `/api/v1/chat/messages?limit=50&before_id=` | 拉取当前 App active session 历史 |
| POST | `/api/v1/chat/turn` | 同步文字 turn |
| POST | `/api/v1/audio/transcriptions` | multipart 上传短音频，返回转写文字 |

### 8.3 关键请求/响应示例

`POST /api/v1/chat/turn`

  "client_message_id": "c_01JZ7P8Y40Q6V2M4"

## 9. 后端改造要求

1. 在渠道能力表中正式增加 `CHANNEL_APP = "app"` 和独立 `__app_active__` scope。
2. App 普通回复全部走同步 HTTP，不调用微信/OpenClaw 网关，不产生 App 主动消息。
3. App 首次聊天不以“有效微信 binding”为资格门槛。
4. 同一手机号已有账号时只读解析并复用主账号；新用户才创建默认账号。
5. 新增 App 历史端点，只读取当前用户主账号的 App active session。
6. 新增 App turn adapter，构造 `ChannelTurnInput` 复用现有 turn 核心。
7. Web/App 幂等顺序按“先查重复 → single-flight → 锁内再查重复 → 限流/执行”实现。
8. App channel binding 的 `chat_id` 保持为空，避免被微信主动消息路由误选。
9. App turn 不改写 legacy `accounts.channel`，也不影响微信渠道的最后活跃时间和主动消息送达窗口。
10. App 禁用提醒、主动消息等需要异步投递的工具，避免生成无法送达的任务。
11. 改造 onboarding 的渠道文案和投递方式；App 首条消息不得被微信专用 welcome 分支吸收。
12. 新增 provider-neutral ASR adapter、音频校验、超时、临时文件清理和成本/延迟指标。
13. 新增会话撤销/续期能力；至少支持当前设备退出，公开版支持注销。
14. Nginx 放行 `/api/v1/*`，配置请求体大小、ASR 超时、聊天长耗时超时和 request id 透传。

## 10. 非功能需求

### 10.1 性能与稳定性

| 指标 | V1 目标 |
| --- | --- |
| App 冷启动到可交互 | 中端真机 P95 ≤ 3 秒（不含首次资源下载） |
| 历史首屏 | 正常网络 P95 ≤ 2 秒 |
| 普通文本回复 | 目标 5–20 秒；超过 20 秒持续显示等待，不能误报失败 |
| ASR | 30 秒以内录音识别 P95 ≤ 5 秒，最终以 provider spike 基线冻结 |
| 文本 turn 成功率 | 排除用户取消/限流后 ≥ 99% |
| Crash-free sessions | 内测稳定后 ≥ 99.5% |
| 隔离 | 任何跨账号、跨 scope 读写均为发布阻断缺陷 |

### 10.2 安全与隐私

- 全链路 HTTPS；生产包禁止信任用户安装的调试证书策略。
- access/refresh token 只存 Keychain/Keystore，不存 AsyncStorage 明文。
- 服务端 token 建议哈希存储并支持撤销、过期和轮换。
- OTP、ASR、chat 分别限流；服务端不信任客户端时长和 MIME 声明。
- 聊天正文沿用现有内容安全流程；ASR 文字发送时同样审核。
- 原始音频不写消息表、不写 debug trace、不用于模型训练，除非未来另行获得明确授权。
- 隐私政策明确手机号、聊天文本、录音临时处理、第三方 SMS/Captcha/ASR 服务商及保存期限。
- iOS `NSMicrophoneUsageDescription` 与 Android `RECORD_AUDIO` 文案和实际用途一致。

### 10.3 可访问性与兼容性

- 登录、验证码、发送和录音控件提供无障碍标签与足够触控面积。
- 支持系统字体缩放，关键按钮不因大字号不可点击。
- 首发最低 iOS/Android 版本、机型矩阵和国内 Android ROM 覆盖在技术 spike 后冻结。
- 网络层兼容 IPv4/IPv6、弱网、切后台和请求中断。

## 11. 数据指标与埋点

埋点只记录事件和性能，不记录手机号明文、聊天正文、ASR 转写或原始音频。

### 11.1 核心漏斗

- `login_view` → `captcha_success` → `otp_send_success` → `otp_verify_success` → `session_created`。
- `chat_view` → `first_message_sent` → `first_reply_received`。
- `voice_entry_tap` → `mic_permission_granted` → `record_complete` → `asr_success` → `voice_transcript_sent`。

### 11.2 关键指标

- OTP 请求、验证和注册/登录成功率。
- 新用户首条消息发送率、首条回复成功率和首回延迟。
- DAU、次日留存、每用户日均有效 turn。
- 消息失败率、幂等命中率、429/5xx/超时占比。
- 麦克风授权率、ASR 成功率、识别延迟、转写后编辑率和放弃率。
- Crash-free、ANR、App 版本分布和强制升级命中率。

## 12. V2 实时语音预留

V1 不实现实时语音，但以下边界必须现在保留：

1. `app` 是独立渠道，不能伪装成 `web`，以便后续声明 realtime、push、TTS 能力。
2. 鉴权支持服务端签发短期 realtime session token，第三方 ASR/TTS/RTC 密钥永不下发长期密钥。
3. 客户端音频层封装为独立模块，页面不直接绑定具体录音库；统一处理权限、音频路由和中断。
4. 后端 ASR 使用 provider adapter，V2 可新增 streaming adapter，而不改用户/账号/审计模型。
5. turn、message、conversation scope 和未来 realtime session 使用独立 id，避免把 HTTP request id 当业务 id。
6. 远端 feature flag 可控制语音入口、provider、最大时长和灰度用户。
7. 预留状态机：`idle → connecting → listening → thinking → speaking → interrupted/ended`。
8. 录音上传代码不被当成实时传输方案；V2 是否使用 WebRTC/WebSocket 需独立技术评审。

## 14. 验收标准

### 14.1 注册/登录

- 新手机号能完成人机验证、OTP、建号和登录，且不产生微信二维码或微信 binding。
- 已有手机号登录不会创建第二个 `platform_user` 或默认账号。
- token 只存安全存储；退出后原 token 无法继续调用受保护接口。
- 过期、错误、重放和过频 OTP 均被服务端拒绝。

### 14.2 文字聊天

- 新用户注册后不经微信即可发送文字并收到文字回复。
- App 消息只写入/读取 `__app_active__` scope，不出现在微信短期历史中。
- 同一 `client_message_id` 连续提交只执行一次并返回同一已持久化结果。
- 超时、断网、429、5xx 下不出现重复用户气泡或重复 AI 回复。
- 两个账号并发使用时，历史、Profile、Memory 和回复绝不串线。

### 14.3 语音输入

- iOS/Android 真机均能完成授权、录音、取消、停止和转写。
- 成功转写只进入输入框；用户确认发送后 AI 返回文字。
- 无人声、空转写和 ASR 失败不会进入 LLM。
- 原始音频不出现在消息历史、普通日志、埋点和长期存储中。
- 60 秒自动停止、来电/后台中断和权限永久拒绝均有可恢复体验。

### 14.4 发布门禁

- P0 用例全绿，SQLite/PostgreSQL 后端回归通过。
- iOS/Android 主链路 E2E 通过，目标机型无阻断级问题。
- 隐私、协议、第三方 SDK、麦克风文案和账号注销满足目标发布渠道要求。
- 生产监控可观察 OTP、chat、ASR、5xx、延迟和 Crash，且不采集正文。
- 支持通过远端开关关闭语音入口或整体进入维护态。

## 15. 实施顺序

1. **M0 决策与 spike**：冻结账号、认证、Captcha、ASR、系统版本和发布范围。
2. **M1 后端 App 最小闭环**：App auth、channel、history、turn、staging 路由与测试。
3. **M2 RN 文字闭环**：登录态、OTP、聊天历史、文字发送、错误恢复。
4. **M3 语音输入闭环**：录音、ASR、编辑、权限与临时音频清理。
5. **M4 合规与稳定性**：设置/注销、埋点、监控、E2E、弱网和真机矩阵。
6. **M5 受控内测**：小流量观察注册成功率、首回、ASR、成本和 Crash 后逐步扩大。

## 16. 主要风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 直接复用 `/web/login` | 新用户被错误引导微信扫码 | 新建版本化 App session 接口，禁止 QR 副作用 |
| App 复用 `web` channel | V2 realtime/push 能力和数据口径混乱 | V1 即引入独立 `app` channel |
| 微信专用 onboarding 被复用 | 欢迎语错误、首条消息被吞、回复误发微信 | App-specific 同步 onboarding 与回归测试 |
| Captcha 在 RN 不稳定 | OTP 主链路阻断 | 原生 SDK/WebView 双方案 spike，保留服务端 provider 抽象 |
| 国内 Android 录音差异 | 部分机型无法录音或格式不兼容 | 真机矩阵、服务端格式探测、客户端录音参数白名单 |
| ASR 成本/延迟超预期 | 语音体验差或成本失控 | 60 秒上限、配额、provider 评测、灰度开关和降级文案 |
| 30 天 opaque Session 暂无轮换 | token 泄漏后的暴露窗口较长 | Keychain/Keystore、服务端哈希与撤销；公开版前评估 refresh/轮换 |
| 同账号跨渠道短期上下文混用 | 用户隐私预期和对话连贯性受损 | 独立 `__app_active__`，长期记忆才共享 |
| 同步 LLM 超时 | 用户看到失败但服务端已完成 | 持久化回复、幂等重试和历史回拉 |
| 无注销/隐私说明 | 无法公开上架或产生合规风险 | 作为公开版发布门禁，不在末期补票 |
