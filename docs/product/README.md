## 2. 权威顺序

发生产品内容冲突时，按以下顺序处理：

1. 隐私、安全、法律与已经冻结的产品宪章原则。
2. 当前目标 PRD 中明确标为已冻结的产品决策。
3. 角色、使命、生命周期、世界内容等产品专题。
4. 视觉原型、交互稿和示例文案。
5. 已实现 V1 基线与历史方案。

技术实现不能自行改变产品规则；产品目标也不能越过真实能力边界。若目标 PRD 与当前实现不同，文档和 UI 必须同时标明“目标状态”和“当前状态”，不能用原型数据伪造已经发生的关系事件。

配套后端文档中明确标记“已冻结”且已有契约与测试支撑的跨端决策，不是普通实现细节。客户端 PRD 与其相反时，应先修订客户端镜像并记录生产/feature-flag 边界，不能把已冻结口径重新降级为待决问题。

## 3. 各层内容边界

### 产品宪章

适合写：产品存在理由、用户承诺、关系模型、体验原则、伦理红线、成功定义和决策取舍。

不适合写：接口、数据库、具体页面、居民数量、邀请码时长、版本排期和临时实验参数。

### 当前目标 PRD

适合写：目标用户、核心场景、版本范围、信息架构、功能规则、数字约束、流程、指标、风险和验收标准。

不适合重复完整人格正文、技术实现细节或运行配置；这些内容应引用专题或技术文档。

### 产品专题

适合写：一个领域内部的完整设计、候选方案、边界、测试场景和版本化资产。例如角色 SOUL、MISSION 和人物差异。

专题必须服从产品宪章，并由当前目标 PRD 决定何时进入版本范围。

### 已实现基线

只回答当前可靠可用的能力和兼容边界。它不是当前目标产品的理念来源，也不因为代码已经存在而自动成为未来产品规则。

## 4. 文档状态约定

| 状态 | 含义 |
| --- | --- |
| 草案 | 可讨论、可大改，不能作为开发承诺 |
| 待评审 | 方案相对完整，但仍有明确待决问题 |
| 已冻结 | 产品决策已经确认；变更需要记录原因和影响 |
| 当前目标 | 当前版本准备交付的产品事实源，可能包含尚未实现能力 |
| 已实现基线 | 已经由代码、接口与测试支撑的现状 |
| 已取代 | 保留历史背景，不再指导新设计与开发 |

每份产品文档头部至少维护：文档角色、状态、版本、最近更新日期、适用范围和关联上位文档。

## 5. 更新纪律

- 产品宪章应少改。变更核心关系模型、增长立场、隐私边界或伦理原则时，必须显式评审，不通过功能 PRD 顺手改写。
- 当前目标 PRD 的数字规则、范围和冻结决策变化时，应更新版本、日期、待决问题和受影响专题。
- 角色与使命使用独立 `persona_version`；已经发布的版本不静默改写，避免既有关系无解释变性。
- 技术文档必须区分现状、已提交未上线、目标设计和未来设想。
- 视觉原型和示例数据不得成为关系状态、角色生命周期和访问权限的事实源。
- 准备 Git 提交时，若改动实质推进海外 AI 陪伴研究或其产品决策，仍需按仓库研究同步规则更新 `docs/research/` 与协作工作区。

## 6. 后续可选拆分

当前先保留一份完整的私人世界 PRD。只有当专题已经稳定且主 PRD 难以维护时，再逐步拆出：

- AI 居民生命周期与离开；
- 私密信箱与新居民相遇；
- 世界内容与发布边界；
- 限时来访与真人会话；
- 产品决策记录目录 `docs/product/decisions/`。

拆分后，当前目标 PRD仍负责版本范围、跨专题流程和最终验收，不变成一组互不相干的功能说明。
