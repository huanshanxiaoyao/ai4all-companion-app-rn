# AI 居民使命设计框架

> 状态：产品与技术衔接讨论稿  
> 更新日期：2026-07-22  
> 目标：使命服务于角色自我与关系连续性，不被“收集 N 个时刻”这一种实现反向限制

## 1. 使命是什么

使命回答“这个居民为什么愿意长期活成现在这个样子”。它不是角色职位、用户任务、留存机制或一句漂亮口号。

一个好的使命同时包含：

- **注意力方向**：角色天然会看见什么。
- **持续行动**：角色愿意反复做什么，但不要求用户签到配合。
- **未解命题**：角色自己也没有标准答案的问题。
- **关系接口**：用户可以怎样参与，又可以怎样不参与。
- **可感知变化**：相处久了，用户能感到使命留下了什么，而不一定是数字增长。

使命应属于角色，但不能把照顾角色使命的责任转嫁给用户。用户不参与、暂时离开或拒绝记录，不应触发内疚、倒计时、降级或关系惩罚。

## 2. 不只一种使命形态

| `kind` 建议 | 使命形态 | 例子 | 合适的进度表达 |
| --- | --- | --- | --- |
| `collection` | 有限收集 | 记下少量真正改变看法的时刻 | `n/N`，允许完成 |
| `practice` | 长期共同实践 | 练习把一天过回身体和感官里 | 最近一次实践、阶段性回顾，不设完成 |
| `inquiry` | 长期开放命题 | 人被理解时，沉默还需不需要解释 | 当前理解、出现过的不同答案，不设完成 |
| `artifact` | 持续生长的共同作品 | 一张可能性地图、一座小院、一本问题簿 | 作品版本、章节、节点或最近变化 |
| `project` | 有阶段的共同计划 | 把一个模糊问题走到可行动的决定 | 里程碑状态，不必计数 |
| `personal_pursuit` | 角色自己的追寻 | 学会分辨离开与自由 | 主要体现在角色表达和世界动态，可无用户进度 |

一个角色可以有一个主使命和一个内在命题，但不应同时运行多个显式进度系统。

### 2.1 推荐的混合结构：长期使命 + 有限章节

非计数使命与阶段成就并不冲突。更适合居民关系的结构是：

```text
长期使命：没有终点，决定角色一直关注什么
  └── 当前章节：一个可以完成的共同产物或小目标
        ├── 若干自然里程碑
        └── 完成后形成一次回顾，但不重置 streak、不自动开启下一轮
```

计数对象应是有内容的共同产物，例如一张问题地图或一种经过验证的生活方法，而不是聊天次数、登录天数、倾诉次数、排盘次数或用户采纳建议的次数。章节完成带来“我们共同做成了一件事”的感受；长期使命继续存在，不因计数结束而失效。

## 3. 现有后端的真实边界

当前后端使命注册表要求每个模板都有 `target_count`；使命工具把进度定义为 `mission_moments` 数量，达到上限后拒绝继续记录；prompt 固定渲染“已记录 n/N 个”。因此只有 `collection` 能无改动落地。

| 实现等级 | 能力 | 改动范围 |
| --- | --- | --- |
| A：当前可用 | 有限收集、固定 `target_count`、追加不可撤销的瞬间 | 只加模板、注册表和测试 |
| A+：计数章节 | 沿用现有计数与封顶，但增加阶段回顾和“第一章已完成”渲染，不再提示继续寻找 | 不改表；小改使命状态渲染、完成态话术和测试 |
| B：小改 | `target_count` 可空；`progress_mode=none/latest/count`；长期实践和开放命题不再显示 `n/N` | mission dataclass、状态渲染、工具返回和测试；现有表可继续复用 |
| C：中改 | 生长型共同作品、可修订摘要、章节或节点、里程碑 | 新增 artifact/milestone 存储与读写接口；保留 account/resident 隔离 |
| D：大改 | 涉及日历、任务、图片、位置或外部行动的长期项目 | 还需权限、工具、失败恢复和跨系统状态，不宜作为首批角色前置 |

### 推荐的最小使命 v2

为支持大部分角色，不必一开始建立复杂任务系统。建议先扩展注册表：

```text
kind: collection | practice | inquiry | artifact | project | personal_pursuit
progress_mode: count | latest | milestone | none
target_count: int | null
completion_mode: finite | ongoing
record_policy: moment | reflection | artifact_entry | none
```

其中 B 级只需要让 `target_count` 可空、移除强制封顶，并按 `progress_mode` 改写 prompt。`mission_moments` 可暂时作为 append-only 的实践/反思记录继续使用，不必立即迁表。

## 4. 候选使命的评审标准

每个候选按以下维度评估：

1. **人物必然性**：换一个角色后是否仍然成立；若成立，使命太通用。
2. **非服务性**：它是否表达角色自己的关注，而不只是“让用户开心”。
3. **非操控性**：不利用完成度、稀缺、离开或用户愧疚维持活跃。
4. **可感知性**：用户能否从对话、回顾或世界动态感到使命存在。
5. **可实现性**：所需状态是否已经有可信事实源；失败时是否会伪造进度。
6. **隐私适配**：记录的是洞察、选择还是原始敏感材料；默认最小化保存。
7. **关系弹性**：用户不想参与时，角色仍能自然生活，关系不塌。
8. **长期空间**：使命不会在几次对话后耗尽，也不会变成重复话术。

## 5. 首批实现建议

- 允许角色使用 `practice`、`inquiry`、`personal_pursuit`，先完成 B 级使命 v2。
- `artifact` 先以 append-only 条目加服务端生成的阶段摘要模拟；验证价值后再做真正可编辑作品。
- `collection` 继续保留，但只在数量本身具有叙事意义时使用，不作为默认模板。
- 首批六位中，小满、砚青、闻溪采用“长期使命 + 有限第一章”：分别形成 12 种生活手感、8 张问题地图和 12 个可能性节点。它们共用现有计数存储，只需 A+ 级完成态渲染，即可验证阶段成就是否真的增加共同成长感。
- 朝颜不统计脆弱表达，司辰与 Aster 不统计求测、排盘、抽牌或所谓命中，避免把情感暴露和命理依赖游戏化。
- App 不常驻显示使命百分比或连续天数。使命首先进入角色上下文，只有自然相关时才向用户表达。
- 多居民上线后使命必须显式绑定角色模板，不能继续按账号哈希随机分配，否则人格与使命会错配。
