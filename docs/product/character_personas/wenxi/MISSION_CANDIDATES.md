### 推荐实现

候选 A 的长期部分是 `artifact + inquiry`，第一版可直接注册为 `collection`：

| 后端字段 | 草案值 |
| --- | --- |
| slug | `twelve_possibility_nodes` |
| display_name | `岔路` |
| statement | `和用户一起为没有标准路线的地图找到 12 个真实可能性节点` |
| target_count | `12` |
| bar | `中：必须是用户真正认可存在的不同路径，并保留状态、现实代价与限制；不要求已经行动` |
| short_label | `可能性节点` |
| inquiry | `离开和自由有什么不同，停留又在什么时候需要勇气` |

阶段回顾建议为 3/6/9/12。当前后端可把每个节点编码为一条结构清楚的文本记录；节点升级必须来自用户事实，不能由 LLM 臆造用户已经行动。达到 12 后显示“第一版地图已形成”，不再提示凑新节点。完整可视地图留到 C 级实现。

`mission_moment.content` 建议使用紧凑文本：`状态 imagined/researched/tried/chosen｜可能性节点｜吸引力｜现实代价与限制`。不保存精确地址、具体行程、账号财务和第三人隐私。

### 完整地图的前置

若验证有效，再增加可修订节点、连接关系、用户可见性和删除能力。现实地点应只存城市级或更模糊信息，默认不保存精确路线；访客不可看到未明确发布的节点。

### 候选 B 的安全边界

实践必须低风险、可拒绝、无连续性：换一首歌、走另一条公共路线、读一个陌生主题都可以。不得变成“每日冒险”、消费推荐或危险挑战。
